Weeks 1–4
Phase 0 — Python foundations
Get fluent enough to build, not just read
▶
Week 1
Core syntax + data types
Variables, strings, lists, dicts, loops, conditionals, functions. Work through Automate the Boring Stuff ch. 1–6. Goal: write a script that reads a file and manipulates its content.
How to learn this week
  1. Set up first (30 min): Install Python 3.12+, VS Code with the Python extension, verify with python --version.
  2. Read Automate the Boring Stuff ch. 1–6 — type every example, don't copy-paste. Muscle memory is the goal.
  3. Lean on your TS knowledge: Variables, loops, and functions map 1:1. Focus on Python quirks: indentation rules, f-strings, list comprehensions, dict methods.
  4. Daily mini-scripts (20 min each): Day 3 — string formatter; Day 5 — list sorter; Day 7 — word frequency counter from a text file.
  5. Capstone: Write a script that reads a .txt file, counts word frequencies, and prints the top 5 words without looking up syntax.
Week 2
Files, modules, packages, virtual envs
Reading/writing files, importing libraries, pip install, creating a venv. Set up VS Code + Python extension. These are the daily tools of every Python AI project.
How to learn this week
  1. Create a real project (Day 1): New folder → python -m venv .venv → activate → pip install requests. Verify the install is isolated to the venv.
  2. TS parallel: venv = isolated node_modules; requirements.txt = package.json; pip install = npm install. The mental model is identical.
  3. Read Real Python's venv primer (realpython.com/python-virtual-environments-a-primer) — the conceptual section explains why isolation matters.
  4. Practice file I/O: Write a script that reads a JSON file, filters some keys, and writes a new JSON file. Use with open() — not a library.
  5. Modularise: Split your week-1 word-counter into utils.py (the function) and main.py (the runner). Import and call it.
Week 3
HTTP, JSON, and working with APIs
Use the requests library to call a public API. Parse JSON responses. Handle errors. This is 80% of what calling LLM APIs looks like.
How to learn this week
  1. Read the requests quickstart (docs.python-requests.org) in one sitting — 15 min.
  2. First call: Hit JSONPlaceholder (jsonplaceholder.typicode.com) or Open-Meteo weather API. Print the full JSON response.
  3. Add error handling: Check response.status_code, call response.raise_for_status(), wrap in try/except. Handle a 404 gracefully.
  4. TS parallel: requests.get(url) ≈ fetch(url); response.json() ≈ res.json(). The model transfers directly.
  5. Build a data fetcher: Collect all pages from a paginated API into a list, save the result to a .json file.
Week 4
Classes, async, and essential AI libraries
Basic OOP. Async/await (LLM streaming uses this). Install and explore: openai, anthropic, httpx, pydantic, python-dotenv, FastAPI.
✓ Milestone: comfortable writing Python from scratch
How to learn this week
  1. Classes (Days 1–2): Write a ChatSession class with __init__, add_message(role, content), and get_history(). This exact pattern appears in every LLM SDK.
  2. Async (Days 3–4): Read Real Python's asyncio guide. Write an async function that fires 3 HTTP calls with asyncio.gather() and compare timing to sequential calls.
  3. Install the AI libraries: pip install openai anthropic httpx pydantic python-dotenv fastapi. Read each README (10 min each) — orientation only, no code yet.
  4. Milestone check: Close your IDE. Write a 30-line script from memory — a class with a method and an async function that calls an API. If you're googling basic syntax, re-read ch. 1–6.
Months 2–3
Phase 1 — API builder
Turn from consumer to builder
▶
Weeks 5–6
First LLM API call + chatbot
Send messages, handle streaming, manage conversation history, understand token limits and pricing. Build a CLI chatbot with a system prompt.
How to learn this week
  1. Day 1 — get your API key: Sign up at platform.openai.com or console.anthropic.com. Store the key in a .env file, load with python-dotenv. Never hardcode keys.
  2. Day 2 — first call: Read the Chat Completions / Messages API docs. Make a single call and print the response. Understand the role/content message format.
  3. Days 3–4 — add streaming: Use stream=True and print tokens as they arrive. Every real chatbot works this way.
  4. Days 5–7 — stateful CLI chatbot: Keep a list of messages, append each user turn, send the full list on every call. Add a system prompt that gives the bot a persona.
  5. Track costs: Log usage.prompt_tokens and usage.completion_tokens. Calculate the cost per call. Build this habit now — it saves money in Phase 3.
Week 7
Prompt engineering fundamentals
Read promptingguide.ai. Zero-shot, few-shot, chain-of-thought. Get structured JSON back reliably from a model.
How to learn this week
  1. Read promptingguide.ai in one session (~2 hours). Take notes on zero-shot, few-shot, and chain-of-thought — these cover 90% of real use cases.
  2. Side-by-side experiment: Pick one task (e.g., "classify this customer message"). Write three versions — zero-shot, few-shot (3 examples), chain-of-thought. Compare outputs.
  3. Structured JSON output: Write a prompt that extracts specific fields from unstructured text. Use response_format: {"type": "json_object"} (OpenAI) or explicit instructions (Anthropic). Validate with Pydantic.
  4. Key insight: Prompts are code. Version-control them, test them, and treat regressions seriously — they can break after a model update without warning.
Week 8
Function / tool calling
Define tools and let the model decide when to call them. Build a weather bot that calls a real API.
How to learn this week
  1. Read the tool use docs for your provider (OpenAI function calling or Anthropic tool use). Understand the 3-turn loop: user → model calls tool → you execute → model responds with result.
  2. Define a tool schema: Write JSON Schema for get_weather(city: string, unit: "celsius" | "fahrenheit"). The description field matters — the model reads it to decide when to call.
  3. Build the weather bot: Mock the actual API (just return fake data). Wire the full loop end-to-end before adding real data.
  4. Add a second tool (e.g., convert_units) and test queries that need the model to choose between them. Watch how description quality affects selection.
  5. TS parallel: This is runtime dispatch based on a function schema — you already think this way from TS function overloads.
Weeks 9–10
Project 1: Smart document summarizer
PDF in → structured JSON summary out, deployed as a REST endpoint on AWS.
✓ Milestone: shipped and deployed project 1
How to learn this week
  1. Design before coding (Day 1): Define your JSON output schema. Write 2–3 test PDFs you'll evaluate against. Know the output contract before writing any code.
  2. Build the pipeline (Days 2–5): Use pdfplumber or pypdf to extract text. Chunk large docs (models have token limits). Call the LLM with a structured-output prompt. Validate with Pydantic.
  3. Wrap in FastAPI (Days 6–7): A single POST /summarize endpoint that accepts a file upload and returns JSON. Your TS backend experience makes this straightforward.
  4. Deploy to AWS (Week 10): Containerise with Docker, push to ECR, deploy to App Runner or Lambda with a Function URL. Use your existing infra knowledge.
  5. Write one eval: Compare the model's output to a hand-written summary for 3 PDFs. Even a simple "does it mention X?" check beats having no eval at all.
Weeks 11–12
Embeddings + fast.ai lesson 1
What embeddings are, how to generate them, cosine similarity. Start fast.ai at 1 lesson/week pace.
How to learn this week
  1. Conceptual grounding first: Read OpenAI's "What are embeddings?" guide. Key insight: a sentence becomes an array of 1536 numbers — semantically similar sentences produce similar vectors.
  2. Hands-on (Days 2–3): Generate embeddings for 10 sentences using openai.embeddings.create(). Compute cosine similarity by hand (dot product / magnitudes). Verify "dog" is closer to "cat" than to "database".
  3. Build semantic search (Days 4–5): Embed a list of 20 FAQs. Embed a user query. Return the top 3 closest FAQs. This is the core primitive behind RAG.
  4. Start fast.ai Lesson 1 (course.fast.ai) — watch it, run the notebook. Goal is orientation, not mastery. One lesson per week from here.
Months 4–6
Phase 2 — Agentic systems
The hottest skill in AI engineering right now

Weeks 13–14
RAG fundamentals
Chunk → embed → store in ChromaDB → retrieve → generate. Build a "chat with your docs" app.
How to learn this week
  1. Build RAG manually first — no LangChain yet. Write each step yourself: chunk text → embed chunks → store in ChromaDB → query → retrieve → prompt LLM with context.
  2. Install: pip install chromadb pypdf sentence-transformers. Use ChromaDB's in-memory mode for fast iteration.
  3. Experiment with chunk size: Try chunk_size=200, 500, 1000 with overlap=50. Print the retrieved chunks for the same query. Chunk size is one of the biggest quality levers.
  4. Build "chat with your docs": Use a document you know well (your own notes, a PDF). Ask questions and verify answers. Wrong answers reveal retrieval failures, not LLM failures.
  5. Key insight: RAG quality = retrieval quality + generation quality. Most failures are retrieval failures — fix retrieval before blaming the model.
Weeks 15–16
LangChain + LangGraph basics
Chains, prompt templates, memory. Build a stateful multi-step pipeline. Learn the concepts more than the library.
How to learn this week
  1. Work through the LCEL quickstart (python.langchain.com/docs/expression_language). The pipe operator pattern — prompt | model | output_parser — is the core abstraction.
  2. Build a chain with memory: Prompt template → model → output parser. Add ConversationBufferMemory. Test with multi-turn conversations.
  3. LangGraph intro: Read the LangGraph "Getting started" page. Build a simple stateful graph with two nodes and an edge. Understand state vs. messages.
  4. Key attitude: LangChain is thin scaffolding. If it's in your way, the underlying API calls are always accessible. Learn the concepts — the library changes frequently.
Weeks 17–18
Agents + tool use
Agent using web search, calculator, and file reader in a loop. ReAct pattern. Handle tool errors and infinite loops.
How to learn this week
  1. Read the ReAct paper abstract (arxiv.org/abs/2210.03629, 15 min). The Reason + Act loop is the mental model behind every agent.
  2. Build a 3-tool agent: Web search (Tavily API, free tier), calculator (a plain Python function), and file reader. Real web search shows you how agents actually behave.
  3. Add error handling: Wrap every tool in try/except and return a descriptive error string. The model can recover from a descriptive error — it cannot recover from an exception.
  4. Add a max_iterations guard: If the agent hasn't finished after N steps, return "stopped: iteration limit reached". Infinite loops in prod are expensive.
  5. Test with hard queries: Ask multi-step questions requiring 3+ tool calls. Most agent bugs appear in how ambiguous intermediate results are handled.
Weeks 19–22
Project 2: Research agent
Autonomous agent that searches, reads, synthesises a structured report. Planner + critic for multi-agent stretch goal.
✓ Milestone: shipped project 2
How to learn this week
  1. Architecture first (Week 19, Day 1): Diagram the planner/executor/critic loop before writing code. Define: input query, intermediate steps, output format.
  2. Use Tavily for real search: Sign up at tavily.com (free tier). Real web search is messier than mocked data — embrace it, it's closer to production reality.
  3. Add a critic step (Week 20): After the agent drafts a report, run a second LLM call: "Is this answer complete? What's missing?" Feed the critique back for one refinement pass.
  4. Stream intermediate steps (Week 21): Show the user "Searching for X…" and "Reading article Y…" as they happen. A UX win that also makes debugging easier.
  5. Write evals (Week 22): Define 5 test queries with expected key facts. Check whether the final report includes them. Fix failures before calling it shipped.
Weeks 23–24
Evaluation frameworks
LLM-as-judge, automated output tests, LangSmith tracing. Evals separate serious AI engineers from hobbyists.
How to learn this week
  1. Set up LangSmith tracing (Day 1): Add the LangSmith SDK to your research agent. Every LLM call now logs inputs, outputs, latency, and token counts automatically.
  2. Write 3 LLM-as-judge evaluators: Relevance (is the answer related to the query?), groundedness (is every claim supported by a retrieved source?), completeness (does it answer the full question?).
  3. Explore Ragas (docs.ragas.io) for RAG-specific evals — faithfulness and answer relevancy come out of the box.
  4. Add to your workflow: Run the eval suite before every deploy. Even 5 test cases catches regressions. This habit is what separates senior AI engineers from prompt engineers.

Weeks 25–26
Local models with Ollama
Run Llama 3 or Mistral locally. Compare quality vs GPT-4. Understand quantization and when self-hosting beats an API.
How to learn this week
  1. Install Ollama (5 min): Go to ollama.com, install, run ollama run llama3. Ask it something. Local inference is now running.
  2. Side-by-side quality test: Write 5 prompts. Run each through Llama 3 8B (local), GPT-4o (API), and Claude Sonnet (API). Document where the gap matters and where it doesn't.
  3. Understand quantization: Try ollama run llama3:8b-q4 vs llama3:8b-q8. Q4 = ~4 GB file, Q8 = ~8 GB. Compare quality on your 5 prompts.
  4. Decision framework: Self-host when data must not leave your machine, volume makes API cost exceed hosting cost, or latency is critical. Otherwise, use the API.
Weeks 27–28
Hugging Face ecosystem
Browse the hub, understand model cards, run inference with Transformers, read benchmark scores meaningfully.
How to learn this week
  1. Create a huggingface.co account and spend 30 min browsing. Filter by task: text-generation, text-classification, feature-extraction (embeddings). Look at download counts.
  2. Run a pipeline (Day 2): pip install transformers torch. Use pipeline("text-generation", model="...") on a model under 2B params (fits in CPU RAM). Print the output.
  3. Read a model card: Pick any top-ranked chat model. Find: training data description, license (critical for commercial use), eval benchmarks.
  4. Benchmark vocabulary: Learn MMLU (general knowledge), HumanEval (code generation), LMSYS Chatbot Arena (human preference). These are the numbers interviewers reference.
Weeks 29–30
Fine-tuning basics
LoRA/QLoRA on a small model on Google Colab (free GPU). Know when fine-tuning beats better prompting.
How to learn this week
  1. Open Google Colab (colab.research.google.com), select a T4 GPU runtime (free). Install: pip install transformers peft trl datasets.
  2. Follow a LoRA tutorial: Search "QLoRA fine-tuning Phi-3 Google Colab" or "LoRA Mistral 7B tutorial". Run the notebook end-to-end before modifying anything.
  3. Use your own data: Create 100–200 examples in instruction/response format. Fine-tune on a task you care about. Dataset quality matters more than dataset size.
  4. Compare fine-tuned vs few-shot: Run 10 test cases through both. Fine-tuning wins on consistent format; few-shot wins on flexibility and diverse inputs.
  5. Decision rule: Fine-tune when you need consistent output style or format. Use prompting when the task changes frequently or dataset is hard to build.
Weeks 31–36
Project 3: Production AI app on AWS
Containerise your research agent, deploy to AWS (ECS + ECR or App Runner), real frontend, logging, cost tracking, eval suite.
✓ Milestone: live production-grade AI app
How to learn this week
  1. Containerise first (Week 31): Write a Dockerfile for your research agent. Build and run it locally. This is your backend background — it should take one day.
  2. AWS deploy (Week 32): Push to ECR, deploy on App Runner or ECS Fargate. Add a domain. Store all secrets in Secrets Manager or Parameter Store.
  3. Add a frontend (Week 33): Streamlit is 20 lines of Python for a working UI. Or use Next.js if you prefer TS. Focus on AI features, not UI polish.
  4. Instrument everything (Weeks 34–35): Log every LLM call: model, token counts, latency, cost per request. Use CloudWatch Logs. Build a simple cost-per-request dashboard.
  5. Eval suite (Week 36): Run your evaluators against the live deployed endpoint. Fix any regressions. This is a production app — treat it accordingly.

Week 1
Big O notation + array manipulation
Time complexity: O(1), O(n), O(n log n), O(n²). Space complexity. Best/worst/average case. Array traversal, in-place modification, and two-index techniques. Your TypeScript Array methods (map, filter, reduce) have real complexity implications — understand them. Solve 10 Easy array problems on LeetCode: Two Sum, Contains Duplicate, Best Time to Buy and Sell Stock.
How to learn this week
  1. Watch first (20 min): search "NeetCode Big O notation" on YouTube — covers everything in the description above. Don't read a textbook for this.
  2. Memorise the 5 complexities: O(1), O(log n), O(n), O(n log n), O(n²). For every loop you write this week, say the complexity aloud before moving on.
  3. Solve Contains Duplicate (#217) first. Brute force = two nested loops = O(n²). Solution: put everything in a Set and check if the size changed. This single problem teaches "trade space for time."
  4. Solve Two Sum (#1): as you walk the array, store each value in a Map<number,number> and check if target - nums[i] is already there. Learn this complement-lookup pattern — it appears in 30% of all problems.
  5. Solve Best Time to Buy and Sell Stock (#121): one pass, carry a running minimum price and a running max profit. Three variables, no nested loop.
  6. After every problem: watch NeetCode's video solution even if you passed. The explanation of why the pattern works is the actual lesson.
✓ Milestone: state time and space complexity for any code you write
Week 2
String manipulation patterns
Immutability, character frequency maps, palindrome detection, anagram checks. TypeScript string methods you already know — now apply them under constraints. Key insight: most string problems are secretly array or hash map problems in disguise. Solve 8 Easy string problems: Valid Anagram, Valid Palindrome, Longest Common Prefix.
How to learn this week
  1. TypeScript gotcha first: string concatenation with += inside a loop is O(n²) due to re-allocation. Build a string[] and call .join('') at the end instead.
  2. For any "count characters" problem, build a Map<string,number> or a 26-slot array (lowercase letters only). This frequency map is the tool for 80% of string problems.
  3. Solve Valid Anagram (#242): two frequency maps and compare, or sort both strings and compare. The sorted approach is simpler to write on a whiteboard.
  4. Solve Valid Palindrome (#125): two pointers converging from both ends, skipping non-alphanumeric characters with a helper like /[a-z0-9]/i.test(c).
  5. Key insight to find: palindrome and anagram problems are both secretly "do these two frequency distributions match?" Once you see that, most string problems become familiar patterns.

Week 3
Hash Maps & Sets — the universal speed-up
When O(n²) brute force becomes O(n) with a map. Frequency counting, complement lookup (Two Sum pattern), grouping (Group Anagrams pattern). TypeScript Map<K,V> and Set<T> are exactly the tools. Solve 10 Medium problems: Group Anagrams, Top K Frequent Elements, Longest Consecutive Sequence.
How to learn this week
  1. Decide Map vs Set before coding: Set = "have I seen this before?" — Map = "what info do I need to remember about this value?" Practice this decision on paper before touching code.
  2. Solve Group Anagrams (#49): use a sorted string as the Map key. All anagrams share the same sorted characters. This is the "canonical form grouping" technique — reappears in many problems.
  3. Solve Longest Consecutive Sequence (#128): add all numbers to a Set. Only start counting from a number with no left neighbour (!set.has(num-1)). This avoids re-counting sub-sequences and gives O(n) instead of O(n²).
  4. Solve Top K Frequent Elements (#347): build a frequency map, then use a bucket array of size n+1 where bucket[freq] holds all numbers with that frequency. Walk backwards to collect the top K. O(n) — better than sorting.
  5. The unifying lesson: a hash map turns every "search for X" from O(n) to O(1). Every time you see a nested loop doing a search, ask if a map eliminates the inner loop.

Two pointers: converging and parallel

How to learn this week
  1. Converging two pointers only work when you can make a greedy argument about which pointer to move. First question always: is the array sorted? If not, can you sort it first?
  2. Solve 3Sum (#15): sort the array, fix one element with an outer loop, run converging pointers on the rest. Critical: skip duplicate values at both the outer loop and the inner pointers to avoid duplicate triplets in your output.
  3. Solve Trapping Rain Water (#42): the insight is that water at position i = min(maxLeft, maxRight) - height[i]. Two pointers track the running maxes as they converge — move whichever pointer has the smaller max (it's the binding constraint).

Sliding window: the O(n) string/array pattern
Fixed-size window vs variable-size window. The expand/shrink pattern. When you need a frequency map inside the window (Minimum Window Substring). TypeScript: natural with left/right index variables. Solve: Longest Substring Without Repeating Characters, Longest Repeating Character Replacement, Minimum Window Substring, Sliding Window Maximum.
How to learn this week
  1. Identify window type before writing any code: fixed (size = k throughout) vs variable (expand until constraint violated, shrink to restore). These have different templates.
  2. Variable window template: let left = 0; for (let right = 0; right < n; right++) { // add nums[right]; while (window violates constraint) { // remove nums[left]; left++; } // update answer; } — write this on paper and keep it visible.
  3. Solve Longest Substring Without Repeating Chars (#3): window = chars in a Set. Violation = duplicate char. Shrink from left until duplicate is gone.

Week 8
Recursion & backtracking
Base case + recursive case. The call stack mental model. Visualising recursion trees. Backtracking as DFS with undo: subsets, permutations, combinations. TypeScript: recursive functions are natural — the shift is trusting the recursive leap. Solve: Subsets, Permutations, Combination Sum, Word Search.
How to learn this week
  1. Watch NeetCode's Subsets solution video before coding anything. The "decision tree" model — at each step you either include or skip an element — makes backtracking visual rather than abstract.
  2. Write the backtracking template on paper: function dfs(start, curr) { result.push([...curr]); for (let i = start; i < nums.length; i++) { curr.push(nums[i]); dfs(i+1, curr); curr.pop(); } } — the .pop() IS the backtracking step.
  3. Solve in order: Subsets (#78) (include/skip), then Combination Sum (#39) (adds reuse — pass i not i+1), then Permutations (#46) (use a used[] boolean array instead of a start index).
  4. Solve Word Search (#79) last — backtracking on a 2D grid. Mark the current cell visited by setting it to '#', recurse in 4 directions, then restore the original character. The restore IS the undo.
  5. Debugging: if your solution gives wrong output, draw the decision tree for a 2-element input and trace exactly which branches your code visits. You are almost always missing the undo step.

Week 9
Binary trees: DFS (preorder, inorder, postorder)
Implement all three traversals recursively and iteratively. Invert a tree, compute diameter, max depth, check balance. The pattern: most tree problems have a 3-line recursive structure — base case, recurse left, recurse right, combine. Solve: Invert Binary Tree, Max Depth, Diameter, Same Tree, Subtree of Another Tree, Balanced Binary Tree.
How to learn this week
  1. Write this template on paper and keep it visible all week: function dfs(node): T { if (!node) return base; const L = dfs(node.left); const R = dfs(node.right); return combine(L, R, node.val); } — most tree problems slot directly into this shape.
  2. Solve Max Depth (#104) first — it IS the template. Base = 0, combine = 1 + Math.max(L, R). Internalize before moving on.
  3. Solve Diameter (#543): the diameter at each node is leftDepth + rightDepth, but the answer is the global max across ALL nodes. Track it with a variable outside the recursion (or an object { max: 0 } passed by reference).
  4. Solve Balanced Binary Tree (#110): return -1 as a sentinel when unbalanced. When any subtree returns -1, propagate it upward without further work — this short-circuits the recursion early.
  5. Key insight: almost every tree problem is postorder. You need both subtree results before computing the current node's value. When stuck, ask: "what should my function return to make the parent's calculation easy?"

Week 10
BFS level-order + Binary Search Trees
Level-order traversal using a queue. Right-side view. BST properties: insert, delete, search in O(h). Validate a BST by passing min/max bounds down the tree — not just comparing to parent. Lowest Common Ancestor. Solve: Level Order Traversal, Right Side View, LCA of BST, Validate BST, Kth Smallest in BST.
How to learn this week
  1. Level-order template: while (queue.length) { const size = queue.length; for (let i = 0; i < size; i++) { const node = queue.shift()!; /* process */ if (node.left) queue.push(node.left); if (node.right) queue.push(node.right); } } — the size snapshot tells you how many nodes are in the current level.
  2. BST validation (#98) — the universal mistake: checking node.val > node.left.val only compares to the direct parent. Instead, pass (min, max) bounds as parameters that tighten as you descend. This handles problems like a right subtree node that's smaller than the root.
  3. LCA of BST (#235): if both targets are less than current, go left; if both greater, go right; if they diverge, current node IS the LCA. Three lines of code once you see the logic.
  4. LCA of general tree (#236): postorder DFS, return the node when it equals p or q. If both left and right recursive calls return non-null, the current node is the LCA.
  5. DFS vs BFS decision rule: use BFS when you need level information (shortest path, right-side view, level averages). Use DFS when you need to aggregate subtree information.

Week 11
Binary search on the answer
Classic binary search first — no off-by-one errors. Then the harder pattern: binary search on an abstract range (find min in rotated array, Koko Eating Bananas). Key insight: if you can write an isValid(k) predicate and valid/invalid regions are monotone, binary search applies. Solve: Binary Search, Find Min in Rotated, Search in Rotated, Koko Eating Bananas, Time Based Key-Value Store.
How to learn this week
  1. Pick one template and use it all week without variation: let l = 0, r = n-1; while (l <= r) { const mid = l + Math.floor((r-l)/2); if (nums[mid] === target) return mid; else if (nums[mid] < target) l = mid+1; else r = mid-1; } — the r-l form prevents integer overflow.
  2. Rotated array logic: first determine which half is sorted. nums[l] <= nums[mid] means the left half is sorted. Then check whether the target falls in the sorted half's range — if yes, search there; otherwise search the other half.
  3. "Search on the answer" template: write isValid(k) first, then wrap it: let l = min, r = max; while (l < r) { const mid = Math.ceil((l+r)/2); if (isValid(mid)) r = mid; else l = mid+1; }. For Koko Eating Bananas: isValid(speed) = can she eat all piles in h hours?
  4. Recognition heuristic: "binary search on the answer" problems often contain the words "minimum maximum" or "maximum minimum." That phrasing signals a monotone valid/invalid split over the answer space.

Heaps & Priority Queues

How to learn this week

How to learn this week

Weighted graphs: Dijkstra and minimum spanning trees
Dijkstra's algorithm (BFS + min-heap). Kruskal's algorithm for MST. These appear less frequently but signal a strong candidate. Solve: Network Delay Time, Cheapest Flights Within K Stops, Min Cost to Connect All Points.
How to learn this week
  1. Dijkstra needs your Week 12 MinHeap. Setup: const dist = new Map([[src, 0]]), heap initialized with [0, src]. Loop: pop minimum, if distance in heap > recorded distance (stale entry) skip it, otherwise relax all outgoing edges.
  2. The key difference from BFS: BFS explores by hop count; Dijkstra explores by cumulative distance. The heap always gives you the closest unfinalized node — that's the greedy choice.
  3. Solve Network Delay Time (#743) first — straightforward single-source shortest path. Answer = max of all final distances. If any node is unreachable (distance = Infinity), return -1.
  4. Solve Min Cost to Connect All Points (#1584) for Prim's MST: similar to Dijkstra but "distance" means cost to connect a point to the current MST, not path from a source. Start from any node, always add the cheapest unconnected point.

Sorting internals

Week 17
1D DP: the Fibonacci family and decision problems

How to learn this week

  2. Solve Climbing Stairs (#70) to get the template. Articulate why: to reach step i you came from i-1 or i-2, so dp[i] = dp[i-1] + dp[i-2]. State the reasoning before writing the code.

2D DP: grids and subsequence problems

How to learn this week
  1. Fill the LCS table by hand before coding. Take "ace" and "abcde", draw the 2D grid on paper, fill it cell by cell. The recurrence will feel obvious once you've done it manually — you'll see the diagonal+1 vs max-of-neighbours logic emerge naturally.
  2. LCS recurrence: if s1[i] === s2[j], dp[i][j] = dp[i-1][j-1] + 1 (extend a shared character). Otherwise dp[i][j] = Math.max(dp[i-1][j], dp[i][j-1]) (skip one char from either string, take the better result).
  3. Solve Edit Distance (#72) next — it's LCS with three operations. At each cell: delete from s1 (dp[i-1][j]+1), insert into s1 (dp[i][j-1]+1), or replace (dp[i-1][j-1] + (s1[i]===s2[j] ? 0 : 1)).

How to learn this week

Week 1
Core SQL — SELECT, WHERE, ORDER BY

How to learn this week

  2. Work through Mode Analytics SQL Tutorial (free): mode.com/sql-tutorial — covers SELECT, WHERE, ORDER BY, and LIMIT in short interactive lessons. Do the first 4 modules.

Week 2
Aggregations — GROUP BY, HAVING, COUNT, SUM, AVG
Turn raw rows into summary metrics. Aggregate functions (COUNT, SUM, AVG, MIN, MAX), GROUP BY to segment, HAVING to filter groups. This is where analysis starts — you're answering business questions like "which region has the most revenue?"
How to learn this week
  1. Key distinction: WHERE filters individual rows before aggregation; HAVING filters groups after aggregation. Write both versions of the same query to feel the difference.
  2. Practice questions: "How many orders per customer?", "What's the average order value by product category?", "Which customers have placed more than 5 orders?"
  3. COUNT(*) vs COUNT(column): COUNT(*) counts all rows including NULLs; COUNT(column) counts only non-NULL values. Run both on a column with NULLs and compare the results.
  4. Build a metrics cheat sheet: Write queries for common business metrics — total revenue, average order value, unique customers, orders per day. You'll reuse these patterns constantly.
  5. Milestone check: Close your browser. Write a query: "What are the top 5 product categories by total revenue, only including categories with more than 100 orders?"
Week 3
JOINs — combining tables
Most real analysis requires more than one table. INNER JOIN (rows in both), LEFT JOIN (all rows from the left, NULLs for no match), RIGHT JOIN, FULL OUTER JOIN. Understand when NULL values appear and what they mean for your data.
How to learn this week
  1. Draw Venn diagrams for INNER, LEFT, and FULL OUTER before writing any code. The visual model prevents 90% of JOIN confusion.
  2. INNER JOIN first: Only rows where the key exists in both tables. Example: SELECT c.name, o.total FROM customers c INNER JOIN orders o ON c.id = o.customer_id.
  3. LEFT JOIN to find gaps: All customers even if they have no orders — orders columns will be NULL. Use to find customers who never ordered: add WHERE orders.id IS NULL after the LEFT JOIN.
  4. Multi-table JOINs: Chain 3 tables (customers → orders → products). Alias tables for readability: customers AS c.
  5. Key insight: In real analytics, LEFT JOIN is the workhorse — you want all records even when related data is missing, and NULLs tell you about data quality problems.
Week 4
Subqueries, CTEs, and CASE WHEN
Subqueries (queries inside queries), CTEs (WITH clauses for readable multi-step queries), CASE WHEN for conditional logic. These make complex analysis readable and maintainable.
✓ Milestone: comfortable writing multi-table SQL with CTEs from scratch
How to learn this week
  1. Start with subqueries: A query inside a WHERE clause: WHERE customer_id IN (SELECT id FROM customers WHERE country = 'USA'). Then move subqueries to the FROM clause as derived tables.
  2. CTEs (WITH clause): Name an intermediate result and reference it like a table. WITH high_value AS (SELECT ... FROM orders WHERE total > 1000) SELECT * FROM high_value JOIN customers .... CTEs are your primary tool for readable multi-step analysis.
  3. CASE WHEN: SQL's if-then-else. Create a bucket column: CASE WHEN total > 500 THEN 'High' WHEN total > 100 THEN 'Medium' ELSE 'Low' END AS order_tier. Then GROUP BY order_tier to see a distribution.
  4. Full pipeline practice: CTE 1 = filter raw data, CTE 2 = aggregate it, CTE 3 = rank the results, final SELECT = top 10 with context. This 4-step shape covers most analyst deliverables.

Weeks 13–14
Descriptive statistics + distributions
Mean, median, standard deviation, variance, IQR, percentiles. Normal distribution and when it doesn't apply. Outlier detection. These are the tools that tell you whether a metric is healthy — not just what the average is, but whether you can trust the average.
How to learn this week
  1. Mean vs median: If your data is skewed (income, order values, page load times), the median is a more honest summary than the mean. Calculate both for a column and compare — when they're far apart, you have skew.
  2. Standard deviation in practice: For a normal distribution, 68% of values fall within 1 standard deviation of the mean. Calculate this for a real dataset column. Values beyond 3 standard deviations are likely outliers.
  3. IQR for outlier detection: IQR = Q3 − Q1. Outliers = values below Q1 − 1.5×IQR or above Q3 + 1.5×IQR. Implement this in pandas and identify outliers in your project 1 dataset.
  4. Plot distributions, don't just compute them: A histogram and a box plot together tell you far more than a mean and standard deviation alone. Always visualize before reporting any metric to a stakeholder.

Statistical inference & A/B testing
Hypothesis testing, p-values, confidence intervals, statistical significance. A/B testing is one of the most important analyst skills at any data-driven company. Understand what a p-value actually means — it's not "the probability you're right." Run a t-test in Python.
How to learn this week
  1. Read "Statistics Done Wrong" (free online): statisticsdoneright.com — short and practical, explains what p-values actually mean and why most analysts misuse them. Read this before touching scipy.
  2. The null hypothesis framework: Assume there's NO difference. Collect data. If the data would be very unlikely assuming no difference (p < 0.05), reject that assumption. p < 0.05 means: "if there truly was no effect, we'd see a result this extreme only 5% of the time by chance."
  3. Run a t-test in Python: from scipy import stats; t_stat, p_value = stats.ttest_ind(group_a, group_b). Practice with synthetic data — generate two groups with known means and verify the test finds the difference.
  4. A/B testing checklist: (1) Define the success metric before the test starts. (2) Calculate the required sample size using a power calculator. (3) Run until you hit that sample size — don't stop early when you see significance. (4) Report confidence intervals, not just p-values.

Weeks 23–24
Advanced SQL — window functions & query optimization
ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM OVER, AVG OVER, PARTITION BY. These separate junior from mid-level analysts. Window functions let you compute running totals, rankings, and period-over-period comparisons directly in SQL — without self-joins.
How to learn this week
  1. Understand the OVER clause: A window function applies to a window of rows defined by OVER. SUM(revenue) OVER (PARTITION BY region ORDER BY date) = running revenue total per region. Key difference from GROUP BY: window functions don't collapse rows.
  2. LAG and LEAD for period comparisons: LAG(revenue, 1) OVER (PARTITION BY product ORDER BY month) gives last month's revenue in the same row as this month's — making MoM growth calculation trivial without a self-join.
  3. Ranking queries: Use ROW_NUMBER in a CTE, then filter WHERE row_num <= 3 to get the top-N per group (e.g., top 3 products per region). RANK vs DENSE_RANK: RANK skips numbers after ties, DENSE_RANK doesn't.
  4. Practice at StrataScratch or Mode: stratascratch.com and mode.com/sql-tutorial/sql-window-functions — both have free window function problems with real company datasets from actual interviews.

Weeks 27–28
Advanced analytics — cohorts, funnels, and retention
Cohort analysis (how do users who joined in month X behave over time?), funnel analysis (where do users drop off?), retention curves. These are the most common analytical questions at product and growth companies.
How to learn this week
  1. Cohort analysis in SQL: Define cohort by first event date. Use DATE_DIFF to calculate periods since first event. Window functions (from week 23) make this clean. Output: a cohort × period table with retention rates — the classic "triangle" chart.
  2. Funnel analysis: For each user, check whether they completed each step. CTEs work well: CTE_step1 = users who did event A, CTE_step2 = those who also did event B. Conversion rate = step N count / step 1 count.
  3. Retention rate formula: (Users active in period N who were also active in period 0) / (Users active in period 0). Day 1, Day 7, Day 30 retention are the standard benchmarks. Day 30 above 20% is considered strong for most consumer apps.
  4. Build a retention curve: Calculate Day 0, 1, 7, 14, 30, 60, 90 retention for a public app dataset. Plot multiple cohorts as lines. The shape tells the story — retention that stabilizes is healthy; retention trending toward zero is a product problem.
